¿Por qué RAG y no solo un LLM?
01 — Motivación

Los modelos como llama3 o qwen saben mucho del mundo general, pero no saben nada de tus documentos: tu reglamento, tus PDFs, tu base interna. RAG une lo mejor de los dos mundos.

Embeddings: convertir texto en números
02 — Fundamento

Para buscar por significado en lugar de palabras exactas, el texto se representa como un vector de ~768 números. Eso es un embedding: una huella numérica del significado semántico.

local_rag/agent.py — _build_retriever()
# nomic-embed-text corre 100% local vía Ollama — sin API key, sin costo por token
from langchain_ollama import OllamaEmbeddings

embedding_model = OllamaEmbeddings(
    model="nomic-embed-text",       # pull con: ollama pull nomic-embed-text
    base_url="http://localhost:11434"
)
vec = embedding_model.embed_query("¿Cómo funciona pgvector?")
print(len(vec))   # 768 — un número por cada dimensión del espacio semántico
print(vec[:4])   # [0.082, -0.431, 0.215, 0.769]

Similitud coseno — cómo PGVector mide qué tan parecidos son dos textos (0 = nada, 1 = idéntico):

0.85

0.61

Chunking: la decisión que más afecta la calidad
03 — Pre-procesamiento

No guardas los documentos enteros en PGVector — los partes en fragmentos (chunks) de tamaño controlado. Un embedding por doc entero es demasiado genérico; uno por chunk es mucho más preciso.

ingest.py — cómo se cargan los docs
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader("documentos/manual_tecnico.pdf")
pages  = loader.load()

splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,    # ~250 palabras por chunk
    chunk_overlap=200,  # 200 chars compartidos entre chunks vecinos
)
chunks = splitter.split_documents(pages)
print(f"{len(chunks)} chunks de {len(pages)} páginas")

# Guarda en PGVector — genera embedding por chunk automáticamente
vector_store.add_documents(chunks)
ParámetroEfectoCuándo ajustar
chunk_size pequeño (400)Chunks precisos pero sin contexto amplio. El LLM puede recibir fragmentos sin coherencia.FAQs, párrafos cortos e independientes.
chunk_size grande (2000)Más contexto por chunk pero el embedding es menos específico.Documentos narrativos, reportes largos, código.
chunk_overlap = 0Riesgo de perder una idea que cae justo en el corte entre dos chunks.Casi nunca recomendado para texto natural.
chunk_overlap = 200Los chunks comparten 200 chars con el vecino — se preserva el hilo de la idea.Punto de partida para la mayoría de documentos.
🎯
El chunking es el paso que más afecta la calidad final. Si el agente da respuestas malas, antes de cambiar el modelo o el search_type, revisa si los chunks tienen sentido por sí solos.
El Retriever y search_kwargs
04 — LangChain

El retriever toma la pregunta, la convierte en vector y busca en PGVector. search_type define la estrategia. search_kwargs son los parámetros que la controlan.

local_rag/agent.py
def _build_retriever():
    vector_store = PGVector(
        embeddings=embedding_model,
        collection_name=config.COLLECTION_NAME,  # nombre de la colección en PostgreSQL
        connection=config.PG_URI,                 # postgresql+psycopg://user:pass@host/db
        use_jsonb=True                           # necesario para filtros por metadata
    )
    return vector_store.as_retriever(
        search_type="similarity",  # ← cambia esto para cambiar el comportamiento
        search_kwargs={"k": 5}
    )

k

Cuántos chunks le llegan al LLM como contexto. Más chunks = más info pero más tokens gastados.

search_kwargs={"k": 5}

filter

Busca solo en docs con cierta metadata. Útil si tienes varias fuentes en la misma colección.

search_kwargs={
  "k": 4,
  "filter": {"source":
    "reglamento.pdf"}
}

score_threshold

Descarta chunks bajo el umbral. Solo aplica con similarity_score_threshold.

search_kwargs={
  "k": 5,
  "score_threshold":
    0.68
}

fetch_k + lambda_mult

Solo con MMR. Candidatos iniciales y balance relevancia vs diversidad.

search_kwargs={
  "k": 5,
  "fetch_k": 20,
  "lambda_mult":0.5
}
similarity — la más simple
05a — Search Type

Calcula la distancia coseno entre el embedding de la pregunta y todos los vectores en PGVector. Devuelve los k más cercanos sin importar qué tan similares sean. Siempre devolverá exactamente k documentos.

Configuración
retriever = vector_store.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}
)

# Siempre devuelve 5, aunque el mejor score sea 0.18
docs = retriever.invoke("¿cuál es la receta del ajiaco?")
print(len(docs))  # 5 — aunque tu base solo tenga docs de Python

✅ Cuándo usarlo

Desarrollo, exploración inicial, validar que PGVector funciona. Es el punto de partida para entender qué tiene tu base.

⚠️ El riesgo

El LLM puede recibir contexto completamente irrelevante y aún así generar una respuesta, lo que produce alucinaciones.

⚠️
Si tu base solo tiene docs sobre Python y el usuario pregunta por "receta de ajiaco", similarity igualmente devuelve 5 docs. El LLM inventará una respuesta con contexto inútil.
🔍
Debug útil: usa similarity_search_with_score(query, k=10) para ver los scores reales de lo que está recuperando tu base. Esos valores te sirven para calibrar el threshold más adelante.
mmr — cuando los chunks se repiten
05b — Search Type

Maximal Marginal Relevance: recupera fetch_k candidatos y luego selecciona los k finales equilibrando relevancia Y diversidad entre ellos.

Configuración
retriever = vector_store.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 5,             # docs finales que recibe el LLM
        "fetch_k": 20,     # candidatos antes de filtrar — mín. k*3
        "lambda_mult": 0.6  # 0=máx diversidad · 1=máx relevancia
    }
)
# Sin MMR: el LLM recibe 5 versiones casi iguales del mismo párrafo
# Con MMR: 5 fragmentos con información distinta y complementaria

lambda_mult = 1.0

Igual a similarity puro. Solo relevancia. Los docs pueden repetir lo mismo.

lambda_mult = 0.6

Balance recomendado. Relevante pero con variedad. Buen punto de inicio.

lambda_mult = 0.0

Máxima diversidad. Puede incluir docs menos relevantes pero muy distintos.

✅
Útil cuando cargaste el mismo reglamento en múltiples versiones o idiomas, o cuando un tema aparece repetido en muchos chunks seguidos del mismo PDF. El LLM recibe contexto variado en lugar de cinco copias de lo mismo.
similarity_score_threshold
05c — Search Type

Como similarity pero con filtro: solo devuelve docs con score ≥ score_threshold. Si ningún chunk supera el umbral, devuelve lista vacía. El agente debe manejar ese caso.

Configuración
retriever = vector_store.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={
        "k": 5,
        "score_threshold": 0.68  # ajusta según los scores reales de tu base
    }
)

docs = retriever.invoke(question)

if not docs:
    # Mejor decirlo que inventar — esto es lo que hace sinrodilla
    return "No tengo información sobre eso en los documentos cargados."
search_type¿Devuelve siempre k?ParámetrosMejor para
similaritySí, siemprekDesarrollo y exploración
mmrSí (hasta k)k, fetch_k, lambda_multChunks repetidos o similares
similarity_score_thresholdNo — puede ser 0k, score_thresholdProducción robusta
🎯
Para ajustar el umbral: usa similarity primero y ejecuta vector_store.similarity_search_with_score(query, k=10). Los scores que veas te dicen dónde poner el threshold con datos reales, no con intuición.
Memoria de sesión: el agente que recuerda
06 — Historial de conversación

Sin memoria, cada pregunta es una conversación nueva. El agente no sabe que hace un momento hablaste de PGVector. RunnableWithMessageHistory envuelve la cadena e inyecta el historial automáticamente.

local_rag/agent.py — gestión de sesiones
# Almacén en RAM — se pierde al reiniciar. Para producción: usar Redis o PostgreSQL
_session_store: dict[str, ChatMessageHistory] = {}

def get_session_history(session_id: str) -> ChatMessageHistory:
    if session_id not in _session_store:
        _session_store[session_id] = ChatMessageHistory()
    return _session_store[session_id]

agent_with_memory = RunnableWithMessageHistory(
    chain,
    get_session_history,
    input_messages_key="input",
    history_messages_key="chat_history"
)

# Cada usuario tiene su session_id — el historial es independiente
response = agent_with_memory.invoke(
    {"input": "¿Cuáles son sus ventajas?"},
    config={"configurable": {"session_id": "usuario_42"}}
)
🔑

session_id

Clave única por conversación. Puede ser UUID, ID de usuario, o en el CLI una clave fija "cli_session". Cada sesión tiene su historial propio.

🔂

Patrón Singleton

El agente se construye una sola vez. Todas las peticiones comparten la misma instancia — y el mismo _session_store.

_agent = None
def get_agent():
    global _agent
    if _agent is None:
        _agent = build_agent()
    return _agent
💬
Sin memoria: "¿Cuáles son sus ventajas?" → el agente no sabe de qué hablas.
Con memoria: tras mencionar PGVector, entiende que "sus ventajas" = ventajas de PGVector.
El prompt de contextualización
07 — Standalone Question

Con historial, preguntas como "¿y el otro?" son ambiguas fuera de contexto. Si se pasan directamente al retriever, busca algo sin sentido. La solución: el LLM reformula la pregunta para que sea autónoma antes de buscar.

"¿y cómo se configura?"
→
LLM reformula
→
"¿Cómo se configura PGVector en LangChain?"
→
Retriever busca bien
local_rag/agent.py — CONTEXTUALIZE_PROMPT
CONTEXTUALIZE_PROMPT = """\
Dado el historial de la conversación y la última pregunta del usuario,
reformula la pregunta para que sea comprensible SIN el historial.
NO respondas la pregunta, solo reformúlala si es necesario.
Si ya es clara, devuélvela tal cual.\
"""

# Solo reformulamos si hay historial — primer turno no lo necesita
if chat_history:
    standalone_q = contextualize_chain.invoke({
        "input": user_input,
        "chat_history": chat_history
    })
else:
    standalone_q = user_input

    config={"configurable": {"session_id": "abc"}})

Los chunks recuperados no son los esperados

# Ver qué recupera y con qué score antes de ajustar parámetros
results = vector_store.similarity_search_with_score("tu pregunta", k=10)
for doc, score in results:
    print(f"{score:.3f} | {doc.page_content[:80]}")
💬
Regla general: si el sistema falla, descarta primero el retriever (con similarity_search_with_score), luego el prompt, luego el modelo. En RAG el 80% de los problemas están en los datos y el chunking, no en el código.
