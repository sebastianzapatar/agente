Search Type en LangChain + PGVector

Cómo usar similarity, mmr y similarity_score_threshold en un sistema RAG con PostgreSQL y pgvector.

Contexto Similarity MMR Threshold Comparación Recomendación

1. Contexto: ¿qué hace un retriever?

En LangChain, un retriever es el componente encargado de buscar documentos relevantes para una pregunta del usuario.

Pregunta
→
Embedding
→
PGVector
→
Documentos relevantes
→
LLM responde
Idea clave: search_type define la estrategia de búsqueda que usará el retriever para seleccionar los documentos. 
retriever = vector_store.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}
)

2. ¿Qué es search_kwargs?

search_kwargs es un diccionario con parámetros adicionales para la búsqueda.

k

Número de documentos

k indica cuántos documentos o chunks quieres recuperar.

search_kwargs={"k": 5}
filter

Filtro por metadata

Permite buscar solo dentro de documentos que cumplan cierta condición.

search_kwargs={
  "k": 5,
  "filter": {"tema": {"$eq": "rag"}}
}
threshold

Umbral mínimo

Se usa con similarity_score_threshold para descartar documentos poco relevantes.

search_kwargs={
  "k": 5,
  "score_threshold": 0.70
}

3. search_type="similarity"

Búsqueda por similitud

Busca los documentos más parecidos semánticamente a la pregunta del usuario.

retriever = vector_store.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 5}
)

¿Qué hace?

Compara el vector de la pregunta con los vectores almacenados en PostgreSQL y devuelve los k documentos más cercanos.

¿Cuándo usarlo?

Cuando estás construyendo una primera versión de RAG y quieres recuperar los documentos más relevantes de manera sencilla.

Ventaja

Es simple, rápido de entender y funciona bien para muchos casos básicos.

Riesgo

Siempre intenta devolver documentos, incluso si la pregunta no tiene relación fuerte con la base de conocimiento.

Ejemplo: si tu base solo tiene documentos de Harry Potter y el usuario pregunta por impuestos, similarity puede devolver documentos aunque no sean realmente útiles.

4. search_type="mmr"

Maximal Marginal Relevance

MMR busca documentos relevantes, pero también intenta que no sean repetidos entre sí.

retriever = vector_store.as_retriever(
    search_type="mmr",
    search_kwargs={
        "k": 5,
        "fetch_k": 20,
        "lambda_mult": 0.5
    }
)

¿Qué hace?

Primero recupera varios candidatos y luego selecciona documentos que combinen relevancia y diversidad.

k

Cantidad final de documentos que se devuelven al LLM.

fetch_k

Cantidad inicial de candidatos antes de aplicar MMR.

lambda_mult

Controla el balance entre relevancia y diversidad. Un valor de 1.0 prioriza relevancia; 0.0 prioriza diversidad.

Úsalo cuando: tus documentos tienen muchos chunks parecidos y quieres que el modelo reciba contexto variado.

5. search_type="similarity_score_threshold"

Filtro por relevancia mínima

Este tipo de búsqueda solo devuelve documentos si superan un umbral mínimo de similitud.

retriever = vector_store.as_retriever(
    search_type="similarity_score_threshold",
    search_kwargs={
        "k": 5,
        "score_threshold": 0.70
    }
)

¿Qué hace?

Busca documentos similares, pero descarta aquellos cuya puntuación esté por debajo del umbral definido.

¿Cuándo usarlo?

Cuando quieres evitar que el agente responda con contexto débil o no relacionado.

Ventaja

Ayuda a reducir alucinaciones en sistemas RAG.

Precaución

Si el umbral es demasiado alto, puede no devolver documentos aunque exista información útil.

Recomendación inicial: prueba con score_threshold=0.70 y ajusta según tus resultados. 
docs = retriever.invoke(question)

if not docs:
    return "No encontré información suficiente en la base de conocimiento para responder."

6. Comparación de los tres search_type

search_type Qué hace Cuándo usarlo Riesgo principal
similarity Devuelve los documentos más parecidos. RAG básico, prototipos, primeras pruebas. Puede traer documentos aunque la pregunta no esté bien soportada.
mmr Devuelve documentos relevantes y diversos. Cuando hay muchos chunks repetidos o parecidos. Puede dejar por fuera algún documento muy similar pero importante.
similarity_score_threshold Devuelve documentos solo si superan un umbral. Cuando quieres controlar alucinaciones y exigir evidencia mínima. Si el umbral es alto, puede no recuperar contexto suficiente.

7. ¿Cuál usar en un proyecto RAG?

Nivel inicial

Usa similarity

Ideal para explicar el concepto a estudiantes y validar que PGVector funciona correctamente.

search_type="similarity"
search_kwargs={"k": 5}
Documentos repetidos

Usa mmr

Ideal cuando los chunks son muy parecidos y necesitas contexto más variado.

search_type="mmr"
search_kwargs={
  "k": 5,
  "fetch_k": 20,
  "lambda_mult": 0.5
}
RAG robusto

Usa similarity_score_threshold

Ideal cuando el agente no debe responder si no encuentra información suficiente.

search_type="similarity_score_threshold"
search_kwargs={
  "k": 5,
  "score_threshold": 0.70
}
Para un trabajo académico: recomiendo empezar con similarity, luego probar mmr, y finalmente usar similarity_score_threshold si el objetivo es evitar respuestas sin evidencia.

8. Código final recomendado

Una configuración robusta para un agente RAG sería:

def get_retriever(vector_store):
    return vector_store.as_retriever(
        search_type="similarity_score_threshold",
        search_kwargs={
            "k": 5,
            "score_threshold": 0.70
        }
    )

Y en el flujo del agente:

docs = retriever.invoke(question)

if not docs:
    answer = "No encontré información suficiente en la base de conocimiento para responder."
else:
    answer = llm.invoke({
        "question": question,
        "context": docs
    })
Mensaje para estudiantes: un buen sistema RAG no solo debe responder; también debe saber cuándo no tiene suficiente información.
